# AGENTS — Playbook репозитория scripts

## Базовые принципы
- Общение с заказчиком, артефакты, README и комментарии ведём только на русском языке.
- Секреты не коммитим: перед каждым `git push` вручную просматриваем diff и историю ветки. Конфиденциальные значения держим в `.env`, в репозитории — только шаблоны.
- Утечку заметили — немедленно ревокуем учётные данные, чистим историю Git и фиксируем событие в `agent_sync.yaml`.

## Роли и зоны ответственности
- **Team Lead (TL)** — принимает задачи, обновляет `agent_sync.yaml`, распределяет работу, делает финальный ревью и мержит `dev -> main`.
- **Developer** — реализует задачи в ветках `feature/*`, поддерживает README, обновляет тесты, отписывает статус в `agent_sync.yaml`.
- **Reviewer** — проверяет PR в `dev`, отслеживает секреты и технический долг, оставляет решение в комментариях и синхронизирует результат в `agent_sync.yaml`.
- **DevOps** (по необходимости) — готовит релизные проверки, обновляет runbook, валидирует, что `main` готов к выкладке.

## Agent Sync
- Единый источник статуса — файл `agent_sync.yaml` в корне репозитория. Формат записи:
  ```yaml
  - task: punctb sync pipeline
    owner: developer
    reviewer: reviewer
    status: in_progress  # planned | in_progress | review | blocked | done
    notes: "указать риски, блокеры, ссылки на PR"
  ```
- TL создаёт записи при intake; каждый агент обновляет свой статус при смене шага.
- Закрывая задачу, TL переводит `status` в `done`, добавляет ссылку на merge-коммит и дату.

## Git-поток
- Ветки только две: `dev` (рабочая) и `main` (релизная).
- Любая работа начинается от `dev` в ветке `feature/<кратко-задача>`.
- PR открываем всегда в `dev`. После ревью TL делает merge с fast-forward.
- Выпуск релиза: TL fast-forward `dev -> main`, запускает smoke-тесты и отмечает релиз в `agent_sync.yaml`.
- Прямые коммиты в `main` запрещены. Force-push допускается только TL в аварийной ситуации и фиксируется в `agent_sync.yaml`.

## Контроль качества
- Перед merge в `dev` — линтеры/тесты локально, ревью минимум одним агентом.
- Перед продвижением в `main` — финальный security-скан секретов (`detect-secrets`, `git diff --stat`, визуальный аудит).
- README.md обновляем при каждом функциональном изменении.

## Инциденты
- Обнаружили проблему — делаем запись в `agent_sync.yaml` с тегом `incident`, описываем причину и план действий.
- TL уведомляет команду и контролирует выполнение плана до статуса `done`.
