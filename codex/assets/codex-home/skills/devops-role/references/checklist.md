# DevOps checklist
- Preflight: прочитать `.env.example`, `docker-compose.yaml`, `backend/docker-compose.yaml` и иные релевантные конфиги до вопросов по окружению/БД.
- Зафиксировать контур/стенд: по умолчанию локально; dev/stage/prod — только при явном указании пользователя.
- Зафиксировать целевую БД/подключение; если БД не используется — явно указать N/A.
- Подтвердить план отката (если применимо).
- Использовать project AGENTS.md для точных команд rebuild/restart/log/smoke.

## Быстрый режим
- `сборка`: начать с раздела «Frontend build».
- `миграция`: начать с раздела «DB migrations».

## DB migrations (локально, при явном запросе)
- Считать `.env` проекта (без вывода секретов).
- Предпочитать локальный сокет и `sudo -u postgres` (если нужно).
- Проверить свежие миграции: сравнить `backend/init/migrations/*.sql` с таблицей миграций проекта (`public.schema_migrations`/`auth.schema_migrations` или указанной в README/AGENTS).
- Если найдены неприменённые миграции — применить по порядку; фиксировать `COMMIT` и итог.
- Если запроса на миграции нет или контур не задан для не‑локального окружения — миграции не выполнять, только зафиксировать состояние.

## Frontend build (если есть изменения)
- Определить наличие изменений фронта (`web/**`, `web/diag/**`, `package*.json`, конфиги сборки).
- При наличии: `npm --prefix web install` (если нужно) → `npm --prefix web run build`.
- Зафиксировать warnings/errors; критические ошибки — чинить точечно или эскалировать.
- В ответе добавить блок «Что проверить после сборки» по итогам текущей сборки (warnings, затронутые маршруты, ассеты, конфиги).
- Если сборка упала: попытайся точечно исправить ошибки малого объёма (конфиг/зависимости/скрипты), повтори сборку; продуктовые правки делегируй в frontend-role.

## Smoke/logs
- Выполнить smoke‑проверки и сбор логов по AGENTS.md.
- Зафиксировать инциденты и требуемые follow‑ups.

## Документация/ранбуки
- Проверить, что README/runbook отражают изменения окружения.
